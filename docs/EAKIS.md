## 一、项目概述与改进方向分析

### 1.1 现有方案核心痛点

通过对原始任务书与流程图的深度分析，识别出以下六大可改进维度：

#### 1.1.1 流程架构层面

- 流程中「数据清洗 → 摘要 → 关键词」路径存在冗余，中间结果未被充分复用于反馈闭环
- Agent 节点编号（1/2/3/4/5/6/7/8/10）存在跳号（缺少 9），建议统一命名为分层职责编码
- 两条主路径（开源情报收集 / 互联网资产采集）缺乏统一的编排调度层，并行能力受限
- 反馈回路（DSL 优化 / 关键词优化 / RAG 知识库更新）未明确触发条件与频率策略

#### 1.1.2 技术实现层面

- LLM 推理为串行调用，关键词提取与资产判定可并行化，预计可压缩整体延迟 40%
- 接口爬取仅依赖 Selenium/Playwright，未引入 CDP（Chrome DevTools Protocol）层精细流量捕获，隐藏接口覆盖率有限
- 漏洞测试引擎对 GraphQL、gRPC 等新型接口协议支持缺失，与目前主流 API 技术栈存在盲区
- 缺乏 Vector DB 持久化存储接口特征嵌入，每次任务重复计算，效率低下

#### 1.1.3 数据模型与知识管理层面

- RAG 知识库仅在报告输出后更新，建议改为增量实时更新（每次清洗后触发片段写入）
- 接口特征库未设计版本控制，靶标系统迭代后的接口变更无法追踪
- 漏洞规则库（接口类型→漏洞类型→测试规则映射）缺乏社区化维护机制和自动扩充策略

#### 1.1.4 安全合规与边界控制

- 自动渗透模块未设计「授权边界」校验机制，存在越界测试法律风险
- 敏感数据（IP、域名、账号等）在 Agent 间传递未明确加密策略和脱敏规则
- 缺少任务熔断与应急回滚机制，单 Agent 异常可能导致整体任务失控

#### 1.1.5 工程化与可观测性

- 日志体系未统一结构化，影响后续的 Agent 行为分析和模型持续训练
- 缺少可视化操作界面（任务管理、Agent 状态监控、结果预览），不适合非技术人员使用
- 模块间依赖关系未通过 API 契约（OpenAPI/gRPC schema）规范化，扩展性差

#### 1.1.6 评估指标体系

- 现有指标（漏爬率、误判率等）均为静态阈值，缺乏动态基准线和置信区间
- 缺少端到端的覆盖率度量（同一靶标不同时间的探测结果对比分析）
- 报告质量评估依赖人工，建议引入 LLM-as-Judge 自动评分机制

## 二、功能描述

### 2.1 系统整体功能架构

系统由六大核心功能模块构成，形成「输入→处理→输出」的完整闭环。每个模块内含多个 Agent，通过统一编排层（Orchestration Layer）协调运行。

| **模块** | **功能描述** | **核心 Agent** | **输出产物** |
| :--- | :--- | :--- | :--- |
| M1 情报采集 | 多源 OSINT 数据自动采集，覆盖新闻/官网/ICP/招投标 | 数据源定义 Agent / 搜索语法生成 Agent / 数据爬取 Agent | 原始情报数据集 |
| M2 关键词引擎 | 基于 LLM 语义分析提取业务/技术/关联主体三类关键词 | 数据清洗 Agent / 摘要 Agent / 关键词生成 Agent | 多维度关键词库 |
| M3 资产发现 | 集成 Fofa/Hunter 检索，AI 多特征融合资产判定 | 资产相关性评估 Agent / DSL 生成 Agent | 精准靶标资产清单 |
| M4 接口爬取 | 大模型浏览器操控，动态交互接口与隐藏接口全面采集 | 智能接口爬取 Agent | 标准化接口特征库 |
| M5 自动渗透 | 接口特征导向的漏洞用例动态生成与自动化执行 | 漏洞测试引擎（规划中） | 漏洞验证报告 |
| M6 报告生成 | LLM 驱动的结构化测试报告自动输出 | 报告输出 Agent | 专业渗透测试报告 |

### 2.2 各模块详细功能说明

#### 2.2.1 M1 情报采集模块

- 支持配置预定义采集规则（行业类型、地域范围、时间窗口）
- 集成 RAG 知识库，历史采集经验反哺搜索语法 DSL 生成
- 多源并行采集：通用搜索引擎（新闻/招投标）+ 空间资产引擎（Fofa/Hunter/Shodan）
- 自动反爬对抗：动态 UA、IP 代理池轮换、随机延迟、浏览器指纹伪装
- 【改进】新增 CDP 层流量捕获，获取浏览器级别的完整 HTTP 流量

#### 2.2.2 M2 关键词引擎模块

- 数据清洗：去重、降噪、核心文本片段抽取（基于 TF-IDF + 领域词库）
- 语义摘要：格式化情报摘要抽取，结构化存储供关键词提取使用
- 三类关键词自动标注：业务关键词 / 技术关键词 / 关联主体关键词
- 关键词权重排序公式：Wk = α×TF-IDF(k) + β×DomainScore(k) + γ×RelevanceScore(k)
- 动态关键词库扩充：同义词库 + 关联规则自动扩充，支持反哺 DSL 生成优化
- 【改进】新增增量式 RAG 片段写入，每次清洗结果实时更新知识库，而非仅在报告后更新

#### 2.2.3 M3 资产发现模块

- 基于关键词批量调用 Fofa/Hunter API，支持分页获取与结果去重
- 五维特征融合判定：IP 归属地 / 域名后缀 / Header 特征 / Body 关键词 / ICP 备案主体
- 余弦相似度匹配算法，阈值 θ=0.75，自动剔除低相关资产
- ICP 备案权重最高，其次 Header 特征，兜底使用页面截图 OCR 识别（favicon hash 比对）
- 【改进】引入 Vector DB（Qdrant/Milvus）持久化资产特征嵌入，支持增量查询

#### 2.2.4 M4 接口爬取模块

- 大模型浏览器操控：LLM 自主识别登录页/注册页，模拟用户点击、输入等交互
- 基于 Few-Shot 学习的动态接口识别，KNN 检索相似接口
- 前端资源全量解析：JS Bundle / HTML Source / Webpack 产物 / Sourcemap
- 接口标准化输出：路径 / 方法 / 参数名 / 参数类型 / 请求头 / 响应结构
- 【改进】新增 CDP 精细流量捕获层，捕获 WebSocket / SSE / gRPC-Web 等非标协议
- 【改进】接口特征库版本控制，支持靶标系统迭代后的接口变更 diff 追踪

#### 2.2.5 M5 自动渗透模块（第二阶段）

- 接口类型分类：查询类 / 操作类 / 文件上传类 / 搜索框类（决策树算法）
- 漏洞规则映射库：接口类型 → 漏洞类型 → 测试规则三层映射
- 动态测试用例生成：结合接口参数格式（JSON/Form-data）自动构造 Payload
- 集成 SQLMap/XSSer 核心能力 + 自定义越权检测插件
- 【改进】新增 GraphQL/gRPC 接口漏洞测试支持
- 【改进】授权边界校验层：所有测试请求必须通过 「授权资产白名单」 二次验证
- 【改进】引入 LLM-as-Judge 二次验证漏洞真实性，降低误报率

#### 2.2.6 M6 报告生成模块

- LLM 自动生成结构化报告，含靶标资产清单 / 接口清单 / 漏洞详情 / 防护建议
- 双格式输出：Markdown（在线预览） + PDF（正式交付）
- 报告生成时间目标：≤ 30 分钟/份
- 【改进】引入 LLM-as-Judge 报告质量评分（准确性 / 完整性 / 可读性三维评估）

## 三、技术栈

### 3.1 总体技术架构

系统采用微服务架构，各模块独立部署为容器化服务，通过消息队列（Kafka）解耦，统一由编排层调度。

| **层次** | **组件/框架** | **用途说明** |
| :--- | :--- | :--- |
| LLM 推理层 | Qwen-7B-Instruct (本地) GPT-4o (云端备用) | 关键词提取、接口识别、报告生成、漏洞验证 |
| Agent 编排层 | LangChain / LangGraph | 多 Agent 协调、工具调用、记忆管理、重试逻辑 |
| 浏览器操控层 | Playwright + CDP Selenium (兼容层) | 动态页面交互、流量捕获、反爬对抗 |
| 数据采集层 | Scrapy / Requests Fofa API / Hunter API | OSINT 多源并行爬取、空间资产检索 |
| 向量存储层 | Qdrant (主) / Milvus (备) RAG: LlamaIndex | 接口特征嵌入存储、情报 RAG 知识库 |
| 关系型存储 | PostgreSQL 15 Alembic (迁移) | 任务管理、资产清单、漏洞记录、报告元数据 |
| 消息队列 | Apache Kafka 3.x | Agent 间异步通信、任务调度、事件驱动架构 |
| 漏洞测试引擎 | SQLMap (改造) XSSer / Nuclei 自研越权检测插件 | 自动化漏洞测试用例执行 |
| 模型服务化 | FastAPI + Uvicorn TorchServe (模型推理) | LLM 和分类模型的 REST API 包装 |
| 任务调度 | Celery + Redis Beat 定时任务 | 异步任务队列、优先级调度、任务状态跟踪 |
| 可观测性 | Prometheus + Grafana OpenTelemetry ELK Stack | 指标采集、分布式追踪、结构化日志 |
| 容器化部署 | Docker + Docker Compose Kubernetes (生产环境) | 服务隔离、弹性伸缩、一键部署 |
| 前端界面 | React 18 + TypeScript Ant Design Pro ECharts | 任务管理面板、Agent 状态监控、报告预览 |
| 模型量化优化 | BitsAndBytes (INT8) AWQ / GPTQ | 降低 LLM 推理延迟，支持多任务并行 |

### 3.2 关键技术选型说明

#### 3.2.1 为何选择 LangGraph 而非 LangChain Agents

- LangGraph 支持有状态图（Stateful Graph）架构，天然适配有反馈闭环的多 Agent 流程
- 内置条件边（Conditional Edge）支持动态路由，无需手动编写 if-else 跳转逻辑
- 人工干预节点（Human-in-the-Loop）可在关键决策点暂停等待人工确认

#### 3.2.2 为何引入 CDP 层

- Selenium/Playwright 的 requests 拦截仅能捕获 XHR/Fetch，无法处理 WebSocket 推送、SSE 流和 gRPC-Web
- CDP 的 Network.getResponseBody 可获取完整响应体，包括 Base64 编码的二进制响应
- CDP 的 Page.addScriptToEvaluateOnNewDocument 可注入 JS 钩子，在页面加载阶段截获 fetch/XHR

#### 3.2.3 Qdrant vs Milvus 选型

- Qdrant：Rust 实现，单节点性能极高，适合 10M 以内向量规模，部署简单（单 Docker 容器）
- Milvus：分布式架构，适合 100M+ 向量规模，但运维复杂度高
- 本项目初期采用 Qdrant，当接口特征向量超过 500 万条时，评估迁移至 Milvus

## 四、核心提示词设计

### 4.1 提示词设计原则

- 结构化输出：所有提示词要求 LLM 输出 JSON 格式，便于程序化解析
- 角色专化：每个 Agent 对应独立的系统提示词，避免多任务混淆
- Few-Shot 示例：每个提示词包含 2-3 个对应场景的输入输出示例
- 边界约束：明确告知 LLM 不允许做什么（如不得修改测试范围、不得生成越权测试）
- 置信度自报：要求 LLM 输出结果时附带置信度分数（0.0~1.0）

### 4.2 关键词提取 Agent 提示词

**系统提示词（System Prompt）**

```text
你是一名专业的网络安全情报分析专家，专精于从开源情报中提取攻击面探测所需的关键词。
你的任务是分析输入的企业情报文本，从中提取三类关键词：
1. 业务关键词：企业主营业务、产品名称、服务类型
2. 技术关键词：使用的技术栈、框架、中间件、数据库
3. 关联主体关键词：子公司名称、合作伙伴、供应商、投资方

输出规则：
- 严格输出 JSON 格式，不得有任何多余文字
- 每类关键词最多 20 个，按相关度降序排列
- 每个关键词附带置信度分数（0.0~1.0）和来源句子索引
- 不得包含通用词汇（如 "公司"、"系统"、"平台"）
